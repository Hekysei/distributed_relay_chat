# Объекты проекта: зависимости и ключевой функционал

Документ описывает основные объекты `distributed_relay_chat` для UML и архитектурного обзора.

## 1) Протокол сообщений и фабрика пакетов

### `Package` (базовый класс пакета)
- Зависит от: базовый тип для `Message`, `TimestampResponse`, `SystemMessage`.
- Предоставляет:
  - сериализацию объекта в JSON (`to_json`);
  - десериализацию JSON в объект класса (`from_json`) с конвертацией дат.

### `Message`
- Зависит от: `Package`.
- Предоставляет:
  - модель пользовательского/системного сообщения чата;
  - установку времени отправки (`set_timestamp_now`).

### `TimestampResponse`
- Зависит от: `Package`, `Message`.
- Предоставляет:
  - подтверждение времени для ранее отправленного сообщения (по `chat` + `message_id`);
  - фабричный метод `from_message` для построения ответа из `Message`.

### `SystemMessage`
- Зависит от: `Package`.
- Предоставляет:
  - транспорт служебных команд/событий (`msg_type`, `body`), например `set_username`.

### `PackageHandler` (интерфейс обработчика)
- Зависит от: типы пакетов (`Message`, `TimestampResponse`, `SystemMessage`).
- Предоставляет:
  - единые точки расширения обработки входящих пакетов: `on_msg`, `on_tsr`, `on_sys_msg`.

### `PackageFactory`
- Зависит от: `PackageHandler`, карта типов `TYPE_CLASS`.
- Предоставляет:
  - определение типа входящего JSON-пакета;
  - создание экземпляра нужного класса пакета;
  - вызов соответствующего обработчика (`process_json`).

### `ActivePackageHandler`
- Зависит от: `PackageHandler`, `ConnectionHandler`, `PackageFactory`.
- Предоставляет:
  - связку сетевого соединения и фабрики пакетов;
  - базовую отправку служебных сообщений и синхронизацию имени пользователя.

## 2) Сеть и транспорт

### `ConnectionHandler`
- Зависит от: `PackageFactory`, модели пакетов (`Message`, `TimestampResponse`, `SystemMessage`).
- Предоставляет:
  - подключение/отключение WebSocket;
  - цикл чтения входящих данных и передачу их в `PackageFactory`;
  - отправку `Message`/`TimestampResponse`/`SystemMessage` в сеть;
  - проверку состояния соединения (`is_connected`).

### `Server`
- Зависит от: `ConnectionHandler`.
- Предоставляет:
  - запуск WebSocket-сервера;
  - создание `ConnectionHandler` на каждое подключение;
  - graceful shutdown (закрытие активных соединений).

## 3) Ядро релея и диспетчеризация

### `Relay` (композиция сервера)
- Зависит от: `Server`, `Dispatcher`, `ProxyDispatcher`, `RelayBot`, `ClientHandler`.
- Предоставляет:
  - сборку всех серверных компонентов;
  - запуск релея;
  - создание обработчика клиента для каждого подключения.

### `DispatchCode` (enum)
- Зависит от: используется в `DispatchResult` и реализациях диспетчера.
- Предоставляет:
  - стандартные коды результата операций диспетчера (успех/ошибка).

### `DispatchResult`
- Зависит от: `DispatchCode`.
- Предоставляет:
  - типизированный результат операции (`ok`, `code`, `params`);
  - форматирование текстовой причины/статуса (`format_error`).

### `DispatcherInterface`
- Зависит от: `DispatchResult`, `Message`.
- Предоставляет:
  - контракт маршрутизации: каналы, подписки, direct, broadсast, модерация.

### `Dispatcher`
- Зависит от: `DispatcherInterface`, `Channel`.
- Предоставляет:
  - хранение каналов и онлайн-пользователей;
  - создание/удаление каналов;
  - регистрацию/удаление пользователей и их callback-ов;
  - broadcast в канал и direct-сообщения между пользователями;
  - базовую валидацию direct-диалогов.

### `Channel`
- Зависит от: `DispatcherInterface`, `Message`.
- Предоставляет:
  - хранение участников комнаты;
  - рассылку сообщений всем участникам (кроме отправителя);
  - уведомления о входе/выходе пользователя из комнаты.

### `UserRole` (enum)
- Зависит от: используется в `AccessRule` и `ProxyDispatcher`.
- Предоставляет:
  - роли пользователя в прокси-диспетчере: `GUEST`, `VERIFIED`, `MODERATOR`.

### `PermissionAction` (enum)
- Зависит от: используется в правилах доступа `ProxyDispatcher`.
- Предоставляет:
  - перечень действий, на которые проверяются права доступа.

### `AccessRule`
- Зависит от: `UserRole`.
- Предоставляет:
  - описание набора ролей, которым разрешено действие.

### `ProxyDispatcher`
- Зависит от: `DispatcherInterface` (декорируемая реализация), `AccessRule`, `make_system_message`.
- Предоставляет:
  - слой авторизации/ролей поверх обычного `Dispatcher`;
  - правила доступа на создание/подписку/рассылку и верификацию;
  - логику единственного модератора;
  - специальную маршрутизацию сообщений модератору.

### `ClientHandler` (серверный обработчик пользователя)
- Зависит от: `ActivePackageHandler`, `DispatcherInterface`, `RelayBot`, `make_system_message`.
- Предоставляет:
  - жизненный цикл одной клиентской сессии на релее;
  - обработку входящих `Message`/`SystemMessage`;
  - маршрутизацию по префиксам чатов (`c/`, `u/`, `m/`, `r/relay`);
  - выдачу `TimestampResponse` и отправку ошибок/системных ответов клиенту.

### `RelayBot`
- Зависит от: `Bot`, `DispatcherInterface`, `make_system_message`.
- Предоставляет:
  - серверные команды (`/create`, `/join`, `/direct`, `/mod`, `/verify`);
  - обратную связь пользователю по результатам операций диспетчера;
  - инициацию служебных сообщений в прямых чатах.

### `make_system_message` (функция)
- Зависит от: `Message`.
- Предоставляет:
  - создание системного сообщения с текущим timestamp.

## 4) Клиентская доменная модель чата

### `Chat` (базовый чат)
- Зависит от: `Message`.
- Предоставляет:
  - хранение истории сообщений;
  - базовое добавление/отправку сообщения в локальную историю.

### `ChatBot`
- Зависит от: `Chat`, `Bot`.
- Предоставляет:
  - чат, где сообщения обрабатываются ботом-командным роутером;
  - добавление набора команд;
  - автоматический вызов команд при отправке текста.

### `RemoteChat`
- Зависит от: `Chat`, `ConnectionHandler`, `TimestampResponse`.
- Предоставляет:
  - отправку сообщений в сеть через `ConnectionHandler`;
  - локальную выдачу `message_id`;
  - синхронизацию timestamp по `TimestampResponse`.

## 5) Клиентская логика и команды

### `Client`
- Зависит от: `ActivePackageHandler`, `ConnectionHandler`.
- Предоставляет:
  - запуск сетевого цикла в отдельном потоке;
  - управление подключением/отключением;
  - базовые действия при успешном соединении (`send_username`).

### `ClientChatBot`
- Зависит от: `ChatBot`, `Client`.
- Предоставляет:
  - локальный сервисный чат `c/client`;
  - команды управления соединением (`/connect`, `/c`, `/d`) и именем (`/name`);
  - приветственные сообщения с подсказками по командам.

### `UserClient`
- Зависит от: `Client`, `Chat`, `RemoteChat`, `ClientChatBot`, типы пакетов.
- Предоставляет:
  - реестр чатов пользователя;
  - создание чатов "по требованию" при получении новых сообщений;
  - обработку входящих сообщений и timestamp-ответов;
  - отправку текста пользователя в активный чат.

### `APPClient`
- Зависит от: `UserClient`, callback-интерфейс UI.
- Предоставляет:
  - callback API для обновления интерфейса (сообщения/добавление/удаление чатов);
  - пользовательские уведомления о состоянии подключения;
  - очистку удаленных чатов после завершения сетевой сессии.

## 6) Командная подсистема и UI

### `FuncArgsPair`
- Зависит от: используется в `CommandRouter`.
- Предоставляет:
  - связку функции команды и ее шаблона аргументов.

### `CommandRouter`
- Зависит от: `FuncArgsPair`.
- Предоставляет:
  - регистрацию команд;
  - синхронную и асинхронную маршрутизацию команды по введенному тексту;
  - разбор позиционных аргументов команды.

### `Bot`
- Зависит от: `CommandRouter`, `Message`, callback отправки.
- Предоставляет:
  - единый интерфейс бота для отправки сообщений;
  - регистрацию и выполнение команд;
  - fallback-ответ при неизвестной команде.

### `TUI_Adapter`
- Зависит от: `UserClient`.
- Предоставляет:
  - текстовый интерфейс (список чатов, окно сообщений, поле ввода);
  - переключение активного чата и отправку пользовательского текста;
  - перерисовку UI при событиях клиента и ресайзе терминала.

### `APP` (точка входа клиента)
- Зависит от: `APPClient`, `TUI_Adapter`.
- Предоставляет:
  - связывание callback-ов клиента с методами UI;
  - запуск приложения и корректное отключение при выходе.
